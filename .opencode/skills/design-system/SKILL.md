## When To Use

Use this skill for visual changes, shared UI components, SCSS changes, responsive behavior, or design token work.

## Styling Rules

- Do not manually edit `app/styles/main.css`; it is generated by `scripts/sync-tokens.js`.
- For design token changes, update the token source used by `scripts/sync-tokens.js`, then run `npm run tokens:sync`.
- Run `npm run tokens:check` when verifying token consistency.
- Follow existing SCSS partial organization in `app/styles/`.
- Shared component styles often sit beside the component as `.scss` files; preserve that convention.
- Keep component filenames globally unique under `app/components/`.
- Maintain the current app visual language unless the task explicitly asks for a redesign.

## UI Implementation

- Reuse base/shared components such as `AppButton`, `AppInput`, `AppEmptyState`, `AppBreadcrumb`, `AppTabBar`, `ProductCard`, and `SectionHeader` where appropriate.
- Check mobile and desktop layouts for page-level UI changes.
- Prefer semantic HTML and accessible labels over div-only UI.
- Keep loading, empty, and error states visually consistent with nearby features.

## Verification

- Styles: run `npm run lint:styles` when SCSS or Vue styles change.
- Tokens: run `npm run tokens:check` after token-related work.
- General changes: run `npm run lint` and targeted tests.
- Build if the change touches Nuxt config, global styles, or broadly used components.
