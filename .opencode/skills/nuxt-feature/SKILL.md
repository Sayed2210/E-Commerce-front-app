## When To Use

Use this skill for customer-facing app work such as home, products, cart, checkout, wishlist, orders, account, auth pages, shared components, Pinia stores, and composables.

## Project Rules

- This is a Nuxt 4 app using the `app/` directory convention.
- `~` and `@` resolve to `app/`.
- Keep component filenames under `app/components/` globally unique because `nuxt.config.ts` sets `pathPrefix: false`.
- Prefer small, local changes that match the existing page/component structure.
- Do not edit `app/styles/main.css`; it is generated by `scripts/sync-tokens.js`.
- Use `useApiClient()` from `app/utils/api.ts` for authenticated API calls and refresh handling.
- Use existing composables before adding new data-fetching logic directly in pages.
- Keep SSR safety in mind: avoid direct browser globals unless guarded by `import.meta.client`, lifecycle hooks, or existing Nuxt utilities.

## Workflow

1. Locate the relevant route in `app/pages/`, component in `app/components/`, composable in `app/composables/`, and tests in `tests/`.
2. Read nearby code before editing so naming, loading state, error handling, and UI patterns match.
3. Put reusable domain logic in composables or stores only when it is reused or meaningfully simplifies the page.
4. Preserve existing SCSS/Tailwind/design token patterns instead of introducing a new styling approach.
5. Add or update focused tests when behavior changes.

## Verification

- Run the smallest relevant test first, for example `npx vitest tests/composables/useProducts.test.ts`.
- For broad changes, follow CI order: `npm run tokens:check`, `npm run lint`, `npm test`, `npm run build`.
- For visual or navigation changes, consider `npm run test:e2e` or a targeted Playwright spec.
